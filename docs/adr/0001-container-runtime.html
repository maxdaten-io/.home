ADR-0001 · 2026-05-24 · jloos · Accepted

Container runtime on Apple Silicon: OrbStack as primary, Colima as OSS fallback

Replace Docker Desktop with a nix-packaged runtime that can build and run linux/amd64 images via Rosetta on Apple Silicon, without sacrificing declarative config.

Context

This repo configures a macOS workstation (Apple Silicon, M-series) declaratively via nix-darwin + home-manager. Container workloads currently rely on Docker Desktop, which presents three friction points:

  • Resource cost — 3–4 GB idle RAM, 30–45 s cold start.
  • Licensing risk — commercial-use limits if employment context changes.
  • Declarative config gap — Docker Desktop installs out-of-band, fragmenting the nix-managed system.

We also need to build and run linux/amd64 images for compatibility with CI registries and upstream images lacking ARM variants. Since Apple opened Rosetta 2 to Linux VMs via Virtualization.framework, x86 Linux binaries now run at ~70–90% native speed inside a Linux VM — a category shift from QEMU userland emulation (5–20× slower).

Decision

Primary Install orbstack via nixpkgs as the daily-driver container runtime.
Fallback Keep colima available as the fully-OSS escape hatch.

Both tools expose a Docker-compatible API, share docker-client and docker-buildx as the CLI surface, and use Apple's Virtualization.framework with Rosetta for amd64. Switching between them requires no application code changes.

Why not Podman?

Daemonless / rootless-first architecture is appealing, but 2025 reports cite Rosetta integration bugs on Apple Silicon. For "run amd64 reliably" it's the weakest of the OSS three. Reconsider if those issues resolve and rootless becomes a hard requirement.

Why not Rancher Desktop?

Not packaged in nixpkgs. Adopting it means a homebrew cask escape hatch — fragmenting the declarative config, the very problem driving this ADR. The k3s integration is nice but doesn't outweigh the loss of single-source-of-truth.

Why not Apple Container (macOS 26+)?

Per-container VM isolation is interesting but Swift-only SDK (no dockerode equivalent), no buildx integration, and binds to a macOS version we can't yet require across all hosts. Revisit when the ecosystem matures.

Consequences

Positive

  • ~15× faster startup, ~10× less idle RAM vs. Docker Desktop.
  • Single declarative install path via nixpkgs.
  • amd64 builds via docker buildx with no extra setup.
  • OSS fallback (Colima) configured — low switching cost.

Neutral / Tradeoff

  • OrbStack is closed-source; binary fetched from upstream, not built from source.
  • One-time GUI permission grant (Virtualization.framework + Rosetta) outside the nix flow.
  • Commercial use requires $8/user/mo — revisit if employment context changes.

Negative

  • Binds container DX to a single vendor with no community fork option.
  • UI features (volumes browser, log viewer) create habits that don't transfer to CI or remote hosts.

Implementation

Add to home.packages in the appropriate module:

home.packages = with pkgs; [
  orbstack
  docker-client     # CLI only; OrbStack provides the daemon
  docker-buildx
];

One-time setup:

  1. Launch OrbStack via Spotlight to grant Virtualization.framework + Rosetta permissions.
  2. Verify amd64: docker run --rm --platform linux/amd64 alpine uname -m → x86_64.
  3. Verify buildx: docker buildx build --platform linux/amd64,linux/arm64 -t test ..

Colima fallback path:

colima start --vm-type vz --vz-rosetta --arch aarch64 --cpu 4 --memory 8
